# Product

## Register

product

## Users

Hackathon judges seeing Living Studio for the first time, the project owner and reviewing engineers. The comprehension target is twenty seconds: what was observed, what GPT proposed, what is unchanged, and what the human can do next.

## Product purpose

Studio visualizes the terminal-first story **install -> observe -> improve**. It renders a validated product map, privacy-minimized workflows, deterministic friction evidence, GPT-5.6's one-file source proposal, the static proof, human approval state and receipts.

The interface must never imply that a prepared proposal is live, that source application proves runtime success, or that Living automatically measured an improvement.

## Brand personality

Honest, calm and evidentiary: a product control room, not a generic AI dashboard. Evidence, model interpretation and authority are visibly distinct.

## Design principles

1. **Truth before beauty.** Show only values and lifecycle state present in the validated input.
2. **Narrative over inventory.** Lead with trigger -> proposal -> proof -> human action; keep hashes available but secondary.
3. **Evidence != interpretation != authority.** GPT can author the proposal but cannot approve or write it.
4. **One obvious next action.** Every lifecycle state explains what changed, what did not and the exact next command/action.
5. **Honest interactivity.** Preview is display-only; disabled controls explain prerequisites.
6. **Legible by default.** Body 14-16px, metadata at least 12px, visible focus and reduced-motion support.

## Accessibility

Target WCAG 2.1 AA: semantic landmarks/headings, keyboard operation, visible focus, `aria-current`, labeled state and tables, reduced-motion alternatives, and responsive reflow to 320px.
