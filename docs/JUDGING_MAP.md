# Judging criteria map

This map turns each live Devpost judging criterion into evidence the repository and demo must expose.

| Criterion | What judges ask | Planned evidence |
| --- | --- | --- |
| Technological Implementation | Is Codex used thoroughly and skillfully? Is the implementation working and non-trivial? | Generated extension diff, capability contract, deterministic proof bundle, test suite, build log, `/feedback` Session ID, install and rollback evidence |
| Design | Is there a complete and coherent product experience rather than only a proof of concept? | Founder Inbox workflow, intent proposal screen, contract review, proof review, explicit approval, installed capability, visible rollback |
| Potential Impact | Does it solve a real and specific problem for a real audience? | Repeated founder workflow, before/after task path, measurable friction reduction, bounded permissions, realistic judge fixture |
| Quality of the Idea | Is it creative, novel, and differentiated? | "Earned evolution" thesis, AI creativity separated from system authority, behavior proof rather than code-only review, reversible capability registry |

## Evidence rule

Every judged claim must point to a runnable behavior, test, commit, or artifact. Planned behavior must remain labeled as planned until it passes the demo smoke test.
